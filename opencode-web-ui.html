OpenCode Quick Start

Get up and running with OpenCode quickly — initialize, configure, and start building

Initialize

Before you start building, configure OpenCode to work with your preferred AI provider and initialize your project.

Connect a Provider

OpenCode works with any LLM provider. Run the /connect command in the TUI, select your provider, and configure your API key. You can use OpenCode Zen (a curated list of tested models) or any other provider you prefer.

aiHackathon Note

For this event, Qwen3.6 has already been set up and is the designated model. Other models are not permitted during the aiHackathon.

Initialize Your Project

Once you've configured a provider, navigate to your project folder and run OpenCode:

Initialize your project by running the /init command. This tells OpenCode to analyze your codebase and create an AGENTS.md file in the project root.

The AGENTS.md file helps OpenCode understand your project structure, coding patterns, and conventions so it can make better suggestions and generate more accurate code.

Remember

Run /init whenever you open a new project or make significant structural changes. Keeping AGENTS.md up to date improves the quality of OpenCode's suggestions.

Usage

Now you're ready to use OpenCode. Talk to it like you would a junior developer on your team — provide enough detail so it understands what you want.

Ask Questions

You can ask OpenCode to explain parts of your codebase. This is especially helpful for areas you didn't write yourself:

How is authentication handled in @packages/functions/src/api/index.ts

Using the @ syntax lets you reference specific files so OpenCode has the right context for its answer.

Add Features

You can ask OpenCode to add new features to your project. For complex features, we recommend using Plan mode first:

  1. Switch to Plan mode using the Tab key (you'll see an indicator in the lower right corner)
  2. Describe what you want to build in detail
  3. Review the plan and provide feedback
▶ Click to reveal example feature prompt 
When a user deletes a note, we'd like to flag it as deleted in the database.
Then create a screen that shows all the recently deleted notes.
From this screen, the user can undelete a note or permanently delete it.

Provide enough details so OpenCode understands the full scope. Talk to it like you're delegating work to a teammate.

Iterate on the Plan

Once OpenCode gives you a plan, you can provide feedback, ask for changes, or add more detail before implementation begins.

You can also include visual references by dragging and dropping images into the terminal. OpenCode will scan the image and use it as a design reference:

We'd like to design this new screen using a layout I've used before.
[Drag and drop image here]
Take a look at this image and use it as a reference.
Visual Reference

OpenCode can process images you provide. Drag and drop screenshots, mockups, or diagrams directly into the terminal to give visual context for your plan.

Build the Feature

Once you're comfortable with the plan, switch back to Build mode by hitting the Tab key again, then ask OpenCode to implement the changes:

Sounds good! Go ahead and make the changes.

OpenCode will analyze the plan and start writing code, creating files, and making changes across your project.

Make Changes

For more straightforward changes, you can ask OpenCode to build directly without reviewing a plan first:

We need to add authentication to the /settings route. Take a look at how this is
handled in the /notes route in @packages/functions/src/notes.ts and implement
the same logic in @packages/functions/src/settings.ts

Again, provide good detail. Reference existing files with the @ syntax so OpenCode understands patterns to follow.

Best Practice

Use Plan mode for complex, multi-file changes. Use direct build mode for simple, localized edits. For more details on spec-driven development, refer to the SDD page in the Training Session.

Undo Changes

Let's say you ask OpenCode to refactor a function, but the result isn't what you wanted. You can undo the changes using the /undo command:

/undo

OpenCode will revert the changes and show your original message again. From here you can tweak your prompt and ask OpenCode to try again:

Can you refactor the function in @packages/functions/src/api/index.ts using a simpler approach?

You can also redo changes using the /redo command if you change your mind:

/redo
Iterate Fearlessly

Don't hesitate to experiment. Use /undo to backtrack, refine your prompt, and try again. Each iteration helps OpenCode understand your intent better.
