Add a Cancel Button

Hands-on exercise: Add a cancel button to the New Post page using Vibe mode

The Problem

Before starting, let's understand what we're fixing:

  1. Open your browser and navigate to the new post form by clicking on the New Post menu item
  2. Notice the current UI - you'll see:
    • A textarea for post content
    • A "Create Post" button
    • No way to cancel or go back
Take a moment

Take a screenshot or mental note of the current UI. After implementing the feature, you'll compare to see what changed.

The issue: Users who change their mind or accidentally click "New Post" have no way to return to the feed without creating a post. This is poor UX - every form should have a cancel option.

Your Task

Add a Cancel button to the New Post page that:

  • Returns the user to the feed
  • Warns them if they've started typing (draft in progress)
  • Matches the existing button styling

Step 1: Start a Vibe Session

  1. Open Kiro (if not already open)
  2. Open the Chat panel (usually on the right side)
  3. Click "New Session" (the + icon at the top of the chat panel)
  4. Select Vibe mode
Why Vibe Mode?

This is a single-component change with simple requirements. Perfect for Vibe mode's conversational approach.

Step 2: Write Your Prompt

Here's a prompt that works well:

Prompt example: "Add a Cancel button to the New Post page. When clicked, it should navigate back to the feed. If the user has started typing, show a confirmation dialog before discarding their draft."

Why this prompt works

Let's break down what makes this prompt effective:

Element What it does Why it matters
"Cancel" button Clear objective Kiro knows exactly what to create
"New Post page" Specific location Kiro knows which component to modify
"navigate back to the feed" Defined behavior Kiro knows what action to implement
"If the user has started typing" Edge case handling Prevents data loss
"show a confirmation dialog" UX detail Specifies how to handle the edge case

What you DON'T need to specify:

  • ❌ File paths (frontend/src/pages/CreatePost.tsx)
  • ❌ Implementation details (React Router's useNavigate hook)
  • ❌ Styling specifics (button classes, colors)
  • ❌ State management (how to track if user typed)

Kiro figures all of this out using your steering documents!

Step 3: Review the Changes

Kiro will analyze the codebase and make changes. When complete:

  1. Look for the "View diff" or "Show changes" button in the chat
  2. Click it to see what changed
  3. Examine the modifications carefully:

Questions to ask while reviewing:

  • Did Kiro modify the correct component? Should be CreatePost.tsx. Check the file path in the diff.
  • Where did Kiro add the button? Is it in a logical location (near the Create Post button)? Does the placement make sense visually?
  • How did it implement the draft detection? Does it check if the textarea has content? Is the logic sound?
  • Does it match the existing button styling? Compare to the Create Post button. Are classes/styles consistent?
  • Are there any unexpected changes? Did Kiro modify anything else? Are all changes related to the cancel button?

Step 4: Test the Feature

Go back to your browser - it should auto-reload. Navigate to the New Post page (http://localhost:5173/new-post)

Testing checklist

Test each scenario systematically:

Empty form test

  • Click Cancel with empty textarea
  • Expected: Navigate immediately to feed (no warning)

Draft in progress test

  • Type some text in the textarea
  • Click Cancel
  • Expected: Confirmation dialog appears

Confirm discard test

  • Type text, click Cancel, click "OK" on dialog
  • Expected: Navigate to feed, text is lost

Cancel discard test

  • Type text, click Cancel, click "Cancel" on dialog
  • Expected: Stay on page, text is preserved

Visual test

  • Does the Cancel button look like it belongs?
  • Is it positioned logically?
  • Does it match the Create Post button style?

What to do if it doesn't work

  • Button doesn't appear? Check browser console for errors (F12 → Console tab). Verify the dev server is still running. Try hard refresh (Cmd+Shift+R or Ctrl+Shift+R).
  • Navigation doesn't work? Check console for routing errors.
  • Styling looks off? Use a follow-up prompt: "Match the Cancel button styling to the Create Post button"

Step 5: Iterate (Optional)

Not happy with something? Use follow-up prompts:

Example refinements:

  • "Make the Cancel button a secondary style with gray background"
  • "Add a cancel icon to the left of the text"
  • "Move the Cancel button to the left of the Create Post button"

What You Learned

  • ✅ Vibe mode is fast - From prompt to working feature in seconds
  • ✅ Steering provides context - You didn't need to explain:
    • What framework you're using (React)
    • Where the component lives (Kiro found it)
    • How to style buttons (matched existing patterns)
    • How to navigate (used the right routing method)
  • ✅ Iteration is natural - Refine with conversational follow-ups
  • ✅ You stay in control - Review diffs before accepting changes

Reflection

Take a moment to consider what just happened:

  • How did Kiro know where to place the button? It used structure.md to understand the codebase organization and found the CreatePost component automatically.
  • What would have happened without steering documents? Kiro would have needed to analyze multiple files to understand your framework (React), routing library (React Router), styling approach, and file structure—consuming context window space before even starting the task. Steering eliminated that overhead.
  • When did you feel Vibe mode was most effective? Likely when you saw how quickly it went from prompt to working code. The conversational nature makes it feel natural, and the iteration cycle is fast.
  • What surprised you? Maybe how Kiro matched the existing button styling without being told, or how it knew to use window.confirm for the dialog, or how it correctly implemented the draft detection logic.

Stretch Goals: Improve the Registration Experience

Remember the password requirements issue from the deployment section? The registration form has two UX problems that are perfect for Vibe mode practice. Try one or both!

Option A: Password Validation

The registration form doesn't tell users what password requirements Cognito enforces, leading to frustrating failed attempts.

Requirements:

  • Display password requirements below the password field
  • Show real-time validation as the user types (checkmarks for met requirements)
  • Disable the Register button until all requirements are met

Prompt example: "Add password requirements validation to the registration form. Show a list of requirements below the password field with checkmarks that update in real-time as the user types. Disable the Register button until all requirements are met."

Why this is a good Vibe exercise:

  • Single component modification (Register page)
  • Clear, testable requirements
  • Real-time state updates
  • Conditional styling and button state

Option B: Better Error Display

When registration fails (e.g., email already exists, password doesn't meet requirements), the app shows a generic "An error occurred" message instead of the actual Cognito error.

Requirements:

  • Display the actual error message from Cognito instead of a generic message
  • Style the error message clearly (red text, visible placement)
  • Clear the error when the user starts typing again

Prompt example: "Improve the registration error handling. Instead of showing 'An error occurred', display the actual error message from Cognito. Style it clearly in red text below the form, and clear it when the user starts typing again."

Why this is a good Vibe exercise:

  • Requires understanding the existing error handling flow
  • Involves try/catch and error state management
  • Tests Kiro's ability to trace through service calls
  • Improves real user experience

The Limits of Vibe Coding

Don't be surprised if Kiro doesn't nail these on the first try. These exercises deliberately push vibe coding to test its boundaries. You might find:

  • The validation UI appears but doesn't update in real-time
  • Error messages are captured but not displayed correctly
  • Some requirements are implemented while others are missed

This is the challenge with vibe coding. As features grow in complexity—requiring coordination across multiple elements, state management, and interconnected logic—a single conversational prompt becomes less reliable. You'll need multiple iterations to get everything right.

This is exactly why Spec-Driven Development exists.

In the next module, you'll learn how to tackle complex, multi-requirement features through a structured approach: requirements → design → implementation. Instead of hoping Kiro understands everything from one prompt, you'll break down the feature systematically, ensuring nothing gets missed.
