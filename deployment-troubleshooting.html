Exercise: Fixing Deployment Scripts

Use Spec mode to diagnose and fix broken deployment configurations in a brownfield scenario

Exercise Overview

In this exercise, you'll use Kiro's Spec mode to troubleshoot and fix deployment issues. Unlike the previous greenfield exercise (adding a commenting feature), this is a brownfield scenario where something is broken and needs systematic diagnosis and repair.

Spec mode is surprisingly effective for troubleshooting because it forces a structured approach: understand the problem, design the fix, and execute methodically.

Common Deployment Issues

Deployment scripts can fail for many reasons. Some common issues include:

  • Environment variable mismatches - Variables referenced in scripts don't match the deployment environment
  • Missing dependencies - Required packages or services aren't installed or configured
  • Incorrect resource paths - File paths or URLs that work locally but fail in production
  • Permission errors - IAM roles or service accounts lacking required permissions
  • Version conflicts - Incompatible dependency versions between development and production

Why Spec Mode for Troubleshooting?

While Vibe mode might seem natural for quick fixes, deployment troubleshooting benefits from Spec mode's structured approach:

Vibe Mode Approach

  • Describe the error, get a suggested fix
  • Risk of addressing symptoms, not root causes
  • No documentation of what was wrong
  • May introduce new issues

Spec Mode Approach

  • Systematic diagnosis of all failure points
  • Documents root cause and fix strategy
  • Creates a task list for methodical repair
  • Leaves a paper trail for future reference

Step 1: Describe the Problem

Open Kiro and switch to Spec mode. Provide a description of the deployment issue:

The deployment pipeline is failing. The build succeeds locally, but the CDK deployment to AWS fails with errors. Please analyze the deployment scripts, identify the root causes, and create a plan to fix them.
Tip

Include any error messages, logs, or symptoms you've observed. The more context you provide, the more accurate Kiro's diagnosis will be.

Step 2: Review the Diagnosis (Requirements)

Kiro will generate a requirements.md that serves as a diagnosis document. It should include:

  • Identification of each failure point
  • Root cause analysis for each issue
  • Impact assessment (what breaks because of each issue)
  • Priority ranking (which issues must be fixed first)

Review the diagnosis carefully. Does it match what you're seeing? Are there issues Kiro missed? Provide feedback to refine the analysis.

Step 3: Review the Fix Strategy (Design)

The design.md document outlines the fix strategy. Look for:

  • Specific changes needed for each deployment script
  • Environment configuration updates
  • Dependency version corrections
  • Testing strategy to verify fixes

Pay special attention to whether the proposed fixes maintain compatibility with your existing infrastructure and don't introduce regressions.

Step 4: Review the Fix Tasks

The tasks.md should break the fixes into ordered, testable steps:

Task 1: Fix environment variable references in deploy.sh Task 2: Update CDK stack configuration for correct region Task 3: Add missing IAM permissions to deployment role Task 4: Fix S3 bucket policy for static asset uploads Task 5: Update package.json dependencies to compatible versions Task 6: Test deployment in staging environment Task 7: Update deployment documentation with new requirements
Important

For deployment fixes, the order matters. Infrastructure changes (IAM, S3) should typically come before application changes. Make sure the task order reflects these dependencies.

Step 5: Execute and Verify

As Kiro executes each fix task:

  1. Review the changes to deployment scripts and configurations
  2. Verify each fix addresses the identified root cause
  3. Test the deployment after each significant change
  4. Document any additional issues discovered during the fix process

Best Practices for Deployment Troubleshooting with Kiro

  • Always test in a non-production environment first - Never apply deployment fixes directly to production
  • Keep a rollback plan - Document how to revert each change if it causes new issues
  • Update your steering documents - If the deployment issue revealed gaps in your tech.md or structure.md, update them to prevent future issues
  • Document the fix - The generated spec documents serve as valuable troubleshooting references for your team

Key Takeaways

  • Spec mode brings structure to troubleshooting, preventing haphazard fixes
  • The diagnosis-design-tasks workflow ensures root causes are addressed, not just symptoms
  • Generated documentation helps the team understand what went wrong and how it was fixed
  • Updating steering documents after fixes prevents the same issues from recurring
  • Spec mode is valuable for both greenfield development and brownfield troubleshooting
