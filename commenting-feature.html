Exercise: Add a Commenting Feature

Use Spec mode to plan and implement a commenting system for the micro-blogging application

Exercise Overview

In this exercise, you'll use TRAE SOLO's Spec mode to add a commenting feature to the micro-blogging application. This is a multi-component feature that touches the frontend UI, backend API, and data model - making it an ideal candidate for Spec-driven development.

Why Spec Mode for This Feature?

Adding comments involves several interconnected changes:

  • Data model - A new Comment entity with relationships to posts and users
  • Backend API - Endpoints for creating, reading, and deleting comments
  • Frontend components - Comment form, comment list, and delete functionality
  • Validation - Input sanitization, length limits, and authorization checks

Trying to implement all of this through conversational Vibe mode would likely result in inconsistent implementations and missed edge cases. Spec mode ensures every aspect is planned before coding begins.

Step 1: Initiate the Spec

Open TRAE SOLO and switch to Spec mode. Provide a high-level description of what you want to build:

Add a commenting feature to blog posts. Users should be able to: - View existing comments on a post - Write and submit new comments - Delete their own comments Comments should display the author name and timestamp.
Tip

Keep your initial prompt focused on the user-facing behavior. TRAE SOLO will use your steering documents to understand the technical context, so you don't need to specify implementation details like "use TypeScript" or "follow REST conventions."

Step 2: Review the Requirements

TRAE SOLO will generate a requirements.md document. Review it carefully and look for:

  • Clear user stories with acceptance criteria
  • Edge cases (empty comments, very long comments, unauthorized deletion)
  • Non-functional requirements (performance, accessibility)
  • Explicit out-of-scope items (nested replies, comment editing, moderation)

If anything is missing or incorrect, provide feedback. TRAE SOLO will revise the requirements until you approve them.

Step 3: Review the Design

Once requirements are approved, TRAE SOLO generates a design.md document. Check that it includes:

  • Comment data model with field types and constraints
  • API endpoint definitions (POST, GET, DELETE)
  • Component hierarchy for the frontend
  • Data flow between components

Pay special attention to how the design aligns with your existing architecture. Your tech.md and structure.md steering documents should guide TRAE SOLO toward consistent patterns.

Step 4: Review the Tasks

TRAE SOLO breaks the implementation into discrete tasks in tasks.md. A well-structured task list might look like:

Task 1: Create Comment model and database schema Task 2: Implement backend API endpoints (POST, GET, DELETE) Task 3: Add input validation and authorization middleware Task 4: Create CommentForm component Task 5: Create CommentList and CommentItem components Task 6: Integrate comments into the Post detail page Task 7: Add error handling and loading states Task 8: Write tests for all new components and API endpoints
Watch for

Tasks that are too large or combine unrelated concerns. Each task should be independently testable and completable. If a task feels too big, ask TRAE SOLO to split it.

Step 5: Execute and Monitor

Once all three documents are approved, TRAE SOLO begins executing the tasks. As each task completes:

  1. Review the generated code changes
  2. Verify the acceptance criteria are met
  3. Approve or request revisions before the next task begins

This iterative approval process ensures quality at each step and gives you the opportunity to course-correct early if something isn't right.

Key Takeaways

  • Spec mode transforms a vague idea into a structured, executable plan
  • The requirements-design-tasks workflow catches issues before they become expensive bugs
  • Steering documents ensure generated specs align with your codebase conventions
  • Iterative task approval maintains quality throughout implementation
  • The generated documents serve as documentation for your team
