Agent Hooks - Automating Your Workflow

Configure automated actions that trigger when TRAE SOLO performs specific operations

Overview

Agent Hooks are automated triggers that execute custom actions when TRAE SOLO performs specific operations. They allow you to enforce coding standards, run tests automatically, generate documentation, and maintain consistency across your codebase without manual intervention.

Think of hooks as guardrails and accelerators - they catch issues before they become problems and automate repetitive tasks that would otherwise slow down your development workflow.

Why Use Agent Hooks?

When working with AI-assisted development, hooks serve several critical purposes:

  • Quality enforcement - Automatically run linters, formatters, and type checkers after TRAE SOLO generates code
  • Consistency - Ensure all generated code follows your team's conventions
  • Safety - Prevent TRAE SOLO from introducing security vulnerabilities or breaking changes
  • Automation - Trigger tests, builds, or deployments after specific operations
Key Principle

Hooks shift quality checks from manual code review to automated enforcement, catching issues the moment they're introduced rather than during a later review cycle.

Types of Hooks

TRAE SOLO supports several hook types that trigger at different points in the development workflow:

Pre-Write Hooks

Execute before TRAE SOLO writes files to disk. Useful for:

  • Validating that proposed changes follow architectural patterns
  • Checking for security-sensitive operations
  • Ensuring required documentation is updated

Post-Write Hooks

Execute after TRAE SOLO writes files to disk. Common uses include:

  • Running code formatters (Prettier, Black, gofmt)
  • Executing linters (ESLint, Pylint, RuboCop)
  • Running type checkers (TypeScript, mypy)
  • Updating generated documentation

Pre-Command Hooks

Execute before TRAE SOLO runs shell commands. Useful for:

  • Validating command safety
  • Setting up environment variables
  • Logging command execution for audit trails

Post-Command Hooks

Execute after TRAE SOLO runs shell commands. Common uses:

  • Processing command output
  • Running tests after build commands
  • Notifying team members of deployment status

Hook Configuration

Hooks are configured in your project's .trae/hooks.json file. Each hook definition specifies:

{ "hooks": [ { "event": "post-write", "pattern": "**/*.ts", "command": "npx eslint {{file}} --fix" }, { "event": "post-write", "pattern": "**/*.py", "command": "black {{file}}" }, { "event": "pre-write", "pattern": "**/*", "command": "check-architecture.sh {{file}}" } ] }

Hook Variables

Hooks support template variables that provide context about the triggering event:

  • {{file}} - The file path that triggered the hook
  • {{files}} - All files modified in the current operation
  • {{event}} - The event type (pre-write, post-write, etc.)
  • {{timestamp}} - When the event occurred

Best Practices

  • Keep hooks fast - Slow hooks disrupt the development flow. Aim for under 5 seconds per hook execution
  • Make hooks idempotent - Running a hook multiple times should produce the same result
  • Fail gracefully - Hooks should report errors clearly without blocking legitimate work
  • Log hook execution - Maintain an audit trail of what hooks ran and their results
  • Test your hooks - Treat hook scripts as production code with proper testing
Important

Hooks that modify files can create infinite loops if not carefully designed. Always ensure your hook commands don't trigger the same hook again.
