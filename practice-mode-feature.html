Add Free Practice Mode

Use Spec-Driven Development to plan and implement a practice mode with unlimited attempts and progress tracking

Exercise Overview

In this exercise, you'll use TRAE SOLO's Spec mode to add a free practice mode to the paraphrasing mentoring app. This is a multi-component feature that requires planning across the data model, API endpoints, and UI components - making it perfect for Spec-Driven Development.

Why Spec Mode?

Adding a practice mode involves:

  • A new data model for practice sessions and attempts
  • API endpoints for starting sessions and recording attempts
  • A PracticeForm component for unlimited paraphrasing practice
  • A PracticeHistory component to display past attempts
  • Progress tracking and statistics display

This complexity makes it ideal for Spec mode's structured approach.

Step 1: Initiate Spec Mode

Open the TRAE SOLO chat panel and switch to Spec mode. Describe the feature you want to build:

Add a free practice mode that allows students to: 1. Start unlimited paraphrasing practice sessions 2. Track each attempt with timestamp and quality score 3. View practice history with progress over time 4. See statistics (total attempts, average score, improvement trend)

Step 2: Review Requirements

TRAE SOLO will generate a requirements.md document. Review it to ensure:

  • User stories cover starting practice, tracking attempts, and viewing history
  • Edge cases are handled (empty submissions, session timeouts, rate limiting)
  • Non-functional requirements include performance targets
  • Out-of-scope items are clearly defined
Example requirements.md: ## User Stories - As a student, I want to start a practice session so I can paraphrase without limits - As a student, I want to see my practice history so I can track improvement - As a student, I want to view statistics so I understand my progress ## Acceptance Criteria - Practice sessions have no daily limit - Each attempt is saved with timestamp and score - History displays attempts in reverse chronological order - Statistics show total attempts, average score, and trend

Step 3: Review Design

TRAE SOLO will generate a design.md document. Review it to ensure:

  • Data model includes PracticeSession and PracticeAttempt entities
  • API contracts define endpoints for session management
  • Component architecture separates form, history, and statistics
  • Database schema changes are documented
Example design.md: ## Data Model ### PracticeSession - id: string (UUID) - userId: string - startedAt: timestamp - status: 'active' | 'completed' ### PracticeAttempt - id: string (UUID) - sessionId: string - originalText: string - paraphrasedText: string - score: number (1-10) - timestamp: timestamp ## API Endpoints - POST /api/practice/sessions - Start new session - POST /api/practice/sessions/:id/attempts - Record attempt - GET /api/practice/sessions/:id/history - Get session history - GET /api/practice/stats - Get user statistics

Step 4: Review Tasks

TRAE SOLO will generate a tasks.md document with ordered implementation steps. Review to ensure:

  • Tasks are ordered by dependencies (data model before API, API before UI)
  • Each task has clear acceptance criteria
  • Test requirements are specified
Example tasks.md: 1. Create PracticeSession and PracticeAttempt models 2. Add API endpoints for session management 3. Create PracticeForm component 4. Create PracticeHistory component 5. Create PracticeStats component 6. Add navigation to practice mode 7. Write integration tests 8. Add error handling and loading states

Step 5: Approve and Implement

Once you approve all three documents, TRAE SOLO will execute the tasks sequentially. You can:

  • Review each task's changes before proceeding
  • Request modifications to any task
  • Pause and test at any point

Step 6: Test the Feature

After implementation, verify:

  1. You can start a practice session from the dashboard
  2. Submitting texts generates paraphrases without limits
  3. Each attempt is saved and appears in history
  4. Statistics update correctly
  5. Error states display appropriately

Key Takeaways

  • Spec mode excels at complex, multi-component features
  • Requirements, design, and tasks provide a clear implementation roadmap
  • Reviewing each document before approval catches issues early
  • The structured approach reduces rework and technical debt
  • Generated documents serve as documentation for the team
