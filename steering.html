Steering - Teaching Kiro about your codebase

Generate foundational steering documents that teach Kiro about your project automatically

Overview

📚 Steering documents provide persistent context about your project, eliminating the need to explain your tech stack and architecture in every prompt. In this section, you'll generate foundational steering documents that teach Kiro about your codebase automatically.

Learning objectives

  • Understand how steering documents improve AI-assisted development
  • Generate automatic steering documents from your codebase
  • Review and customize steering content for your project needs

What are steering documents?

Steering documents are markdown files that capture key information about your project:

  • product.md - Application purpose, user personas, and features
  • tech.md - Technology stack, architecture patterns, and conventions
  • structure.md - Codebase organization and key files

These documents are stored in .kiro/steering/ and automatically referenced in every Kiro conversation, providing consistent context without consuming your conversation window.

How steering differs from traditional documentation

Traditional documentation is written for humans to read when they need information. Steering documents are written for AI to reference automatically in every interaction.

Traditional Docs

  • Stored in /docs or wikis
  • Read when needed
  • Often becomes outdated
  • Requires developers to remember to check

Steering Docs

  • Stored in .kiro/steering/
  • Automatically loaded by Kiro
  • Generated from codebase analysis
  • Always available in AI context

Example: What's in each foundation steering file

  • product.md - Application purpose and features
  • tech.md - Technology stack and patterns
  • structure.md - Codebase organization

Understanding context windows

AI agents like Kiro have a context window - a finite amount of information they can process in a single session. Think of it like short-term memory: the agent can only "see" a certain number of tokens (roughly words) at once.

Why this matters:

  • Every file Kiro reads consumes context window space
  • Analyzing your codebase to understand tech stack and patterns uses valuable context
  • Less space remains for your actual code changes and reasoning

Example: If Kiro needs to understand your React + TypeScript + AWS setup, it might read:

  • package.json (dependencies)
  • tsconfig.json (TypeScript config)
  • Several component files (to understand patterns)
  • Infrastructure files (to understand AWS setup)

This could consume some of the context window before even starting your task.

Why steering matters

Steering documents solve the context window problem by providing persistent knowledge that's always available:

Without steering documents, Kiro would need to analyze your entire codebase for every request to understand:

  • What framework and language you're using
  • Where different components are located
  • What patterns and conventions to follow

With steering documents, this essential information is pre-loaded without consuming valuable context space, allowing Kiro to focus on your actual request and use the context window for the code that matters.
