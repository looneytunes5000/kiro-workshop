Spec-Driven Development

Transform high-level ideas into structured specifications with requirements, design, and implementation tasks

Overview

Spec-Driven Development is Kiro's structured approach to building complex features. Unlike Vibe mode's conversational style, Spec mode formalizes the software development process by generating three key artifacts before writing any code: requirements, design, and tasks.

This approach ensures that complex features are thoroughly planned, reducing rework and technical debt that often accumulates during rapid prototyping.

When to Use Spec Mode

Spec mode excels at:

  • Complex, multi-file features requiring architectural decisions
  • Features that need clear requirements and acceptance criteria
  • Cross-cutting changes affecting multiple system components
  • Features requiring design documentation for team alignment

Spec mode is not ideal for:

  • Simple, single-file changes (use Vibe mode instead)
  • Quick bug fixes or typos
  • Exploratory prototyping where requirements are unclear
Key Principle

Research shows that addressing issues during the planning phase is 5 to 7 times less costly than resolving them during implementation. Spec mode bakes this principle into your AI-assisted workflow.

The Spec Workflow

When you initiate a Spec in Kiro, the agent guides you through a structured process:

1. Requirements
2. Design
3. Tasks
4. Implementation

Step 1: Requirements

Kiro generates a requirements.md document that captures:

  • User stories with acceptance criteria
  • Functional and non-functional requirements
  • Edge cases and error conditions
  • Out-of-scope items to prevent scope creep

Step 2: Design

Kiro generates a design.md document that includes:

  • Architecture decisions and rationale
  • Data model changes
  • API contracts and interface definitions
  • Sequence diagrams for complex flows

Step 3: Tasks

Kiro generates a tasks.md document that breaks the implementation into:

  • Discrete, ordered implementation steps
  • Clear acceptance criteria for each task
  • Dependencies between tasks
  • Test requirements

Step 4: Implementation

Once you approve all three documents, Kiro executes the tasks sequentially, updating progress as it goes. You can review and approve each task before it proceeds to the next.

Vibe vs Spec: Choosing the Right Mode

Vibe Mode

  • Conversational, interactive
  • Best for simple, focused changes
  • No documentation generated
  • Fast iteration
  • Relies on steering documents for context

Spec Mode

  • Structured, document-driven
  • Best for complex, multi-file features
  • Generates requirements, design, tasks
  • Thorough planning upfront
  • Creates audit trail for the team

How Steering Documents Enhance Specs

Your steering documents provide critical context that makes Spec mode more effective:

  • product.md - Ensures requirements align with existing features and user personas
  • tech.md - Guides design decisions to match your technology stack and patterns
  • structure.md - Ensures tasks respect your codebase organization
Important

The quality of your steering documents directly impacts the quality of generated specs. Well-maintained steering documents lead to more accurate requirements, better design decisions, and cleaner task breakdowns.
